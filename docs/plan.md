# План разработки

## Цель проекта
Polka это веб-приложение для ведения личной библиотеки, заметок и публичных подборок книг. Пользователь должен получать не просто список книг, а личное книжное пространство: с отслеживанием статусов чтения, отзывами, цитатами, рекомендациями и возможностью делиться частью библиотеки по ссылке.

Проект делается как демонстрация инженерного уровня в Angular и смежной full-stack разработке. Поэтому важны не только сценарии интерфейса, но и архитектура приложения, изоляция доменов, строгая типизация, тестируемость, производственный CI/CD и предсказуемая доставка на VPS через контейнеры.

## Пользовательские сценарии
1. Пользователь регистрируется, входит в систему и попадает на свою публичную страницу-библиотеку, где сразу видит статистику, списки и коллекцию книг.
2. Пользователь добавляет книгу через полноценную форму: загружает обложку, заполняет метаданные, статус, рейтинг, заметки и цитаты, после чего видит книгу в своей библиотеке.
3. Пользователь фильтрует библиотеку по статусу и жанру, сортирует книги, отслеживает сколько прочитал за период и открывает карточку книги с мнением и цитатами.
4. Пользователь создаёт публичный список рекомендаций, выбирает книги из библиотеки и делится ссылкой на подборку или на свою библиотеку с гостями.

## Риски
1. Несоответствие версий Angular 21, Taiga UI 4 и вспомогательных библиотек — снижать через раннюю инициализацию проекта, фиксацию версий и проверку сборки в CI.
2. Разрастание frontend-логики вокруг библиотек, списков и публичных представлений — снижать через явную доменную модель на backend и тонкие UI-store/signal-store слои.
3. Перегрузка главной страницы из-за изображений обложек, статистики и бесконечной ленты — снижать через lazy loading, оптимизацию изображений, skeleton/loading states и измерение bundle size.
4. Расхождение между приватным и публичным режимами профиля — снижать через единые policy-checks на backend и отдельные e2e/component сценарии для owner/guest.
5. Срыв итерационного графика из-за слишком крупных веток — снижать через roadmap, мелкие feature-ветки, Conventional Commits и MR по одному завершённому вертикальному срезу.
