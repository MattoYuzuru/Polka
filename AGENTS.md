# AGENTS.md

## Общие правила работы
- Перед началом новой задачи проверь актуальность ветки: `git fetch --all --prune` и `git pull --ff-only`.
- Нельзя работать напрямую в `main` или `master`. Для каждой задачи создавай отдельную ветку с понятным именем, например `feat/auth-shell` или `chore/ci-bootstrap`.
- В конце работы обязательно:
  - прогоняй релевантные тесты, линтеры и сборку;
  - обновляй документацию, если изменилось поведение, архитектура или процесс;
  - делай `git push` в свою рабочую ветку.

## Качество и стиль
- Коммиты оформляй по Conventional Commits.
- Не оставляй код без типизации в публичных API без отдельного обоснования.
- Бизнес-логику по возможности держи на стороне backend, а не frontend.
- Комментарии в коде пиши на русском языке и только там, где без них сложнее понять намерение.

## Обязательные проверки перед завершением задачи
- `npm run lint` для frontend, если проект уже инициализирован.
- `npm run test` для frontend unit-тестов, если задача затрагивает логику.
- `npm run test:ct` или релевантные Playwright component tests, если меняется пользовательский сценарий.
- `go test ./...` для backend, если меняется серверная логика.
- `docker compose config` для изменений в инфраструктуре и контейнерах.

## Документация и трекинг
- Держи в актуальном состоянии `docs/roadmap.md`: отмечай выполненные этапы, текущую итерацию и ближайшие шаги.
- Если меняется архитектурное решение, обновляй соответствующие документы в `docs/`.
- Новому агенту должно быть достаточно открыть `docs/roadmap.md`, чтобы понять текущую фазу проекта.
